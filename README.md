# SoundStream Neural Audio Codec

Реализация нейросетевого аудио-кодека на основе статьи:
Zeghidour, N., Luebs, A., Omran, A., Skoglund, J., & Tagliasacchi, M. (2021). SoundStream: An End-to-End Neural Audio Codec. arXiv:2107.03312.

## Ссылки на датасеты

1. Для обучения использован датасет LibriSpeech, часть train-clean-100  
2. Для подсчета метрик использован датасет LibriSpeech, часть test-clean

Ссылка на полный датасет LibriSpeech с нужными подразделами в Kaggle:
[LibriSpeech на Kaggle](https://www.kaggle.com/datasets/a24998667/librispeech)

## Структура репозитория

data/dataset.py           — предобработка датасета LibriSpeech

model/encoder_decoder.py  — архитектура энкодера и декодера

model/rvq.py              — Residual Vector Quantizer

model/discriminator.py    — Multi-Scale Wave + STFT дискриминатор

model/losses.py           — losses, используемые в нашей архитектуре

training/train.py         — скрипт обучения

inference.py              — загрузка аудио и его прогон через кодек

evaluate.py               — подсчёт метрик STOI и NISQA

demo.ipynb                — демо для удобной загрузки аудио по `url`-ссылке
и его прогон через кодек, с удобным дисплеем и комментариями по использованию (подразумевается запуск в среде Google Colab)

## Отличие реализации от статьи

Следование статье было не полным, поэтому ниже указаны некоторые тонкости реализации:

1. Мы изменили параметры входного аудио с 24Гц на 16 Гц, а следовательно и некоторые численные параметры архитектуры для согласованности с этим изменением, подробнее в комментариях к реализации модели. Наиболее заметное изменение мы сделали в strides сверток для энкодера и декодера, вместо [2, 4, 5, 8] используем [2, 4, 5, 5]. (Обусловлено заданием.)

2. Мы изменили функцию активации в обоих дискриминаторах на `LeakyReLU(0.2)`. (Обусловлено заданием.)

3. Мы не используем `FiLM` в энкодере и декодере. (Обусловлено заданием.)

4. В `RVQ`(`Residual Vector Quantizer`) мы не делаем `EMA` (`Exponential Moving Average`). Мы делаем обычный `back propogation` (В задании не просили реализовывать `RVQ` в точности как в статье, а просили написать свою реализацию.) Так же мы не делаем `Enabling bitrate scalability with quantizer dropout`.(Обусловлено заданием.)

5. В целом `RVQ` не включает в себя многие детали из статьи, например, мы не подчищаем не исопользуемые векторы. Или мы инициализиуем кодбук рандомно, а не с помощью каких-то усреднений первого батча. Можно сказать, что это просто самая базовая реализация. Более подробно смотрите в комментариях к `rvq.py`.

6. Обучение происходит на датасете LibriSpeech train-clean-100, а подсчет метрик на LibriSpeech test-clean. Полные ссылки для скачивания датасетов указаны выше. То есть обучение было не на чистых и зашумленных данных, а только на чистых. (Обусловлено заданием.)

7. В статье явно не описаны некоторые детали, например, shortcut в `ResidualUnit2D`, который пришлось сделать, чтобы размерности согласовывлись. Кажется, что это классическая практика при работе с residual connection, так как мы должны складывать выход и вход, которые могут оказаться разного размера.

8. В декодере есть тонкость с размерами входов. Либо это статья подразумевает, что это на самом деле количество выходов (так как декодер в некотором смысле "зеркальное отображение" энкодера), либо я не доразобралась, почему это не работает так, как хотелось бы. Возможно, мною из вида было упущено еще какое-то преобразование, на основании которого вход должен был быть больше, чем написано в моей реализации.

## Воспроизведение обучения

Обучение проводилось на Kaggle $T4 \times 2$. Датасет train-clean-100 подключался через интерфейс Kaggle (Add data) и был доступен по пути `/kaggle/input/datasets/a24998667/librispeech/train-clean-100`. В зависимости от того, как именно вы будете его подтягивать в Input (по ссылке на датасет, указанной выше, или по какой-то другой), вам нужно перепроверить путь.

```python
!python training/train.py \
  --data_dir /путь/до/train-clean-100 \
  --use_comet \
  --comet_apikey YOUR_COMET_APIKEY \
  --comet_project YOUR_PROJECT_NAME \
  --hf_repo YOUR_HUGGING_FACE_REPO_NAME \
  --hf_token YOUR_HUGGING_FACE_REPO_TOKEN \
  --save_dir NAME_OF_DIR_FOR_CHECKPOINTS_IN_HF_REPO \
  --resume /путь/до/нужного/чекпоинта
```

Разумеется, некоторые из этих аргументов опциональны.

`--data_dir` - единственный обязательный аргумент. Путь до train-clean-100.

Если вы хотите логировать метрики в свой аккаунт CometML

`--use_comet` - вы должны указать, что вы вообще используете CometML

`--comet_apikey` - cюда передается ваш API для CometML

`--comet_project` - название вашего проекта в СometML, по умолчанию назовется "soundstream"

Если вы хотите:

(i) логировать в HuggingFace промежуточные checkpoints и финальную модель, то

`--hf_repo` - указываете название своего репозитория в HF (желательно создать его специально для этого проекта)

`--hf_token` - токен, сгенерированный вами в вашем репозитории HF

(ii) дополнительно к пункту (i) продолжить прерванное обучение с помощью одного из уже сохранившихся checkpoint

`--resume` - указываете путь до последнего checkpoint, который у вас сохранился ЛОКАЛЬНО (об этом ниже)

## Про использование готовой модели и дообучение с --resume

В большинстве скриптов путь до датасета подразумевает ЛОКАЛЬНЫЙ путь. То есть вам нужно скачать checkpoint локально или обучить и сохранить checkpoint локально самостоятельно.
Скачать checkpoint с HF можно, например, вот так:

```python
from huggingface_hub import hf_hub_download
checkpoint_path = hf_hub_download(
    repo_id="Finikita255/SoundStream-checkpoints",
    filename="checkpoints/checkpoint_final.pth",
)
```

В обсуждаемом HF репозитрии Finikita255/SoundStream-checkpoints лежат так же промежуточные checkpoints. В случае, если вы будете обучать с нуля с использованием логгирования в HF, то в вашем указанном репозитории они тоже сохранятся и вы сможете продолжить обучение с помощью аргумента `--resume`. Однако и здесь в качестве пути вы должны будете передавать локальный путь. Вы совершенно аналогично можете скачать локально свой checkpoint с HF для продолжения обучения.

## Воспроизведение подсчета метрик

Подсчет метрик проводился на Kaggle $T4\times 2$. Датасет test-clean подключался через интерфейс Kaggle (Add data) и был доступен по пути `/kaggle/input/datasets/a24998667/librispeech/test-clean`. В зависимости от того, как именно вы будете его подтягивать в Input (по ссылке на датасет, указанной выше, или по какой-то другой), вам нужно перепроверить путь.

Далее, для подсчета метрик запускается скрипт:

```python
!python evaluate.py \
  --checkpoint /путь/до/checkpoint_final.pth \
  --test_dir /путь/до/test-clean \
  --n_files 2620 \
  --compute_nisqa \
  --comet_apikey YOUR_COMET_APIKEY
```

Некоторые из этих аргументов опциональны.

`--checkpoint`  - локальный путь до checkpoint (обязательно).

`--test_dir` - путь до test-clean в вашем окружении (обязательно, удобнее всего подгрузить на Kaggle)

`--n_files` - количество файлов, на которых вы хотите подсчитать метрики и их среднее. Если вы хотите прогнать весь датасет test-clean, то указывайте 2620, по умолчанию стоит значение 50.

`--compute_nisqa` - если вы хотите считать не только STOI, но и NISQA (внутри подсчета nisqa ленивый импорт, чтобы в случае с нежеланием считать эту метрику вы по минимуму столкнулись с конфликтом версий)

Если вы так же хотите залогировать метрики в CometML, то вам нужно так же указать свой API от аккаунта CometML: `--comet_apikey`.
